Open Source Development Lifecycle and related useful Git Commands







Hey reader, I am happy to share with you the experience of taking part in open source contributions.

 Please read the following if you want to understand what goes into Open Source Development Lifecycle (for example, with Github account).

 The first thing any open source enthusiast could do is searching for open source code repositories.

 Then you can look into the Issues tab of repositories.

 After going through the issue, please choose an issue that you can work on comfortably given your skillset. 

 The next thing you can do is to fork repository online via GitHub, to your own GitHub account

 After forking, please clone the repository to the local machine, using Github desktop.

 Just to get an idea of the project, please open the repository folder in IDE - VSCode or Intellij.

 Please check developer-documentation README.

 Most of the repositories contain a README for developers so that contributors get an idea, on how to set up the project (maven/Gradle) in the local machine
how to set up the database locally, how to run the docker image locally
how to run the tests locally, how to build the code, 

 This README file helps developers, to a great extent,  to focus productively on the issue which they want to contribute. And, from experience, it saves a lot of time too.

 Now comes the step where you can do the setting up of dependencies for the project.

 For this, please check the README file to follow the steps and set up dependencies.

 Now you can enjoy searching for relevant source files to make changes.
Put your detective goggles and start checking for variable names/ class names/method names/ service names/ DTO objects / HTML component files, which relates to the issue you are working on.
And now happily find a solution that addresses the issue.
Make changes in source code with great interest and dedication.
Test the solution locally while sipping tasting coffee.

 Now to the most interesting part, deploy the solution locally and based on setup provided in README, you can run the server-side (mvnw, in most cases for web dev) and run the client side (yarn start, in most cases for web dev).
This gives you the chance to verify your solution manually.

 Once the solution is working, please adapt the test cases for the modified code ( this step seems to be very important in most of the cases).
The quality of the project is maintained based on the adaptation of test cases, absolutely.

 After updating test cases, please run all the test cases - both the client side and server-side. This ensures that later Travis CI build succeeds.

 Now you have patiently reached the stage to commit changes locally to your repository.

 Please push changes to the remote repository, using GitHub
Once changes are pushed, go to the original repository and you can create a pull request.
Once the pull request is created, GitHub will automatically show whether changes can be merged.
Sometimes, depending on the repository, pull requests are automatically sent for Travis build.

You can comment on the pull request, notifying maintainers that changes can be merged.
Once the pull request is seen by the maintainer, the maintainer accordingly merges the changes, if it solves the issue.
Now the issue corresponding to this commit is closed by the maintainer, along with the pull request.
Congratulations on your contribution to open source.
As you go through the above phases, you might be interested in following git commands

-- get a list of any configured remote URLs for your repository\ with the command. 

 ( sometimes you might want to check which remote repositories, your local repository is linked to )


git remote -v

Courtesy: StackOverflow
-- updating a forked repository 

 sometimes, after forking a repo and working on your own, other contributors might have contributed and now, the parent repo is ahead of your local repo, so you need to keep it updated. because your local repo is pointing to your remote ( your own GitHub repo ) and not actual parent remote repo. so following commands will help

 

# Add the remote, call it "upstream":

git remote add upstream https://github.com/whoever/whatever.git

# Fetch all the branches of that remote into remote-tracking branches,
# such as upstream/master:

git fetch upstream

# Make sure that you're on your master branch:

git checkout master

# Rewrite your master branch so that any commits of yours that
# aren't already in upstream/master are replayed on top of that
# other branch:

git rebase upstream/master


Courtesy: StackOverflow
-- remove files from git commit

 Sometimes, by mistake, you might have committed a file and even created a pull request, now you want to remove that file from the commit.  ( It happens, for example, there is a file, which is supposed to be ignored from commit but is not added to ignore list, and you accidentally committed it. And you realize it later )
  1. remove the file git rm
  2. commit with amend flag: git commit --amend

Courtesy: StackOverflow
