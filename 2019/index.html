    graph: breadth first traversal ( java )


    courtesy: https://media.geeksforgeeks.org/wp-content/uploads/bfs-5.png

    1. Problem is that we want to traverse this graph, to visit all the vertices.

     We are going to use : breadth first traversal approach

     Here, we take one vertex, go to all its neighbours.. reach dead end. Then go to another vertex.. go to all its neighbours.. so on.

     For this we maintain queue. so if one vertex is done with all its neighbours, we remove it from queue and move on to the next element in the queue.

     The logic is as follows:

    1. We somehow have to maintain data about the graph - its vertices and edges.
    2. For this , adjacency list representation is being used.
    3. We will leverage this adjacency list, so that we can iterate over each vertex's list ( neighbours of a given vertex )
    4. Adding an edge to a graph, means that, we are basically adding an element to the adjacency list of a particular vertex.
    5. While we do BFS, we want to maintain an important variable: which is boolean array of visited vertices, so that we dont want to end in some infinite loop, given that a vertex can point to itself and there could be cycles also between vertices. ( cyclic )

    This code is also shared in github: BreadthFirstSearchGraph

    Below rendered code, is brought to you by: HILITE.ME

    ( apology for the analogy commented in the code :) )


    package main.geeksforgeeks.graph;

import java.util.LinkedList;
import java.util.ListIterator;

/**
 * See documentation of this class code in your IDE, to visualize the graph being used in this problem
 * <img src="https://media.geeksforgeeks.org/wp-content/uploads/bfs-5.png" />
 *
 * Problem : https://www.geeksforgeeks.org/breadth-first-search-or-bfs-for-a-graph/
 * Useful article to understand BFS and DFS difference: https://techdifferences.com/difference-between-bfs-and-dfs.html
 */

public class BreadthFirstSearchGraph {


    // this needs to somehow store vertices and edges
    // we shall use adjacency list to store, let's use linkedlist

    int numVertices;

    LinkedList<Integer> adjacencyList[];

    public BreadthFirstSearchGraph(int p_numVertices) {
        numVertices = p_numVertices;
        // so, for visualization purpose, we are going to have a list of total number of vertices. one list to cover this.


        adjacencyList = new LinkedList[numVertices];
        // now comes the tricky part, if you have not understood adjacency list concept.
        // for each of this vertex, we are going to maintain another list of reachable vertices from this vertex.
        // so, 4 vertex means 4 lists, that's why we are doing the below initialization.

        for (int i = 0; i < numVertices; i++) {
            adjacencyList[i] = new LinkedList<>();
        }
    }

    // we need a function , to add edges to the graph
    public void addEdge(int a, int b) { // this will add an edge from a to b
        // the assumption is that, for the vertex a, already a linkedlist is created, which could hold b
        adjacencyList[a].add(b);
    }

    // for BFS, we are going to ultimately print out vertices reachable from the input vertex
    // since we are allowing 2 vertices to follow each other / directed edges,
    // we are going to create a boolean vector which will mark the visited vertex, to disallow
    // multiple visits to same vertex
    void BFS(int vertex) {

        boolean visited[] = new boolean[numVertices];
        visited[vertex] = true; // we are visited the input vertex initially, from there, we will go visit others

        // let's create a queue of vertices which we are able to visit from the vertex, starting with vertex
        // itself. Please note, for these kind of graph problems, it is always better to draw a graph and visualize
        // the problem.

        // we want a data structure to hold the vertices, so that we will iterate over each of these
        // vertices, to visit reachable vertices. for this, let's take a linkedlist
        LinkedList<Integer> queueOfVertices = new LinkedList<>();
        queueOfVertices.add(vertex);

        while (queueOfVertices.size() != 0) {

            Integer a_vertex = queueOfVertices.poll();
            // now we can print that, we have indeed reached this vertex within the graph.
            System.out.println(a_vertex);
            // get the adjacency list of this a_vertex, iterate over them and add them to queueOfVertices
            // apparently, for linkedlist, we have a different kind of iterator, which is listiterator
            ListIterator<Integer> adjacencyListIterator = adjacencyList[a_vertex].listIterator();

            // please note here, that we are taking 1st level neighbours of a vertex ( meaning directly reachable)
            // so, we are going breadth-wise.. taking one level at a time..
            // take a vertex, visit all its neighbours... then go to one neighbour, visit all its neighbours.. so on.


            // whereas in dfs, what we will do is: take a vertex, visit its neighbour.. go to that neighbour now itself.. visit its neighbour..
            // now move on to the next neighbour now only, visit its neighbour.. till dead-end.
            // then go back to the first vertex..

            // so, what we can realize that, in first vertex, it might have other multiple neighbours unvisited so far..
            // but in bfs, we just finish off visiting all neighbours of a vertex at one time, then go to its neighbours
            // and do it..



            while (adjacencyListIterator.hasNext()) {

                Integer reachableVertex = adjacencyListIterator.next();
                // now i have reached this vertex, so, let's mark that we have indeed visited it.
                // one point to note that, we might have already visited this vertex in the past, given
                // that directed edges are allowed in some types of graph ( directed graph, cyclic? )

                if (visited[reachableVertex] != true) {
                    //to disallow multiple visits to same vertex
                    visited[reachableVertex] = true;
                    queueOfVertices.add(reachableVertex);
                }

                // if you want to experiment something, see infinite loop in action, comment out
                // if block above.. you will realize importance of visited array.

            }


        }

    }


    public static void main(String[] args) {
        BreadthFirstSearchGraph g = new BreadthFirstSearchGraph(4);

        g.addEdge(0, 1);
        g.addEdge(0, 2);
        g.addEdge(1, 2);
        g.addEdge(2, 0);
        g.addEdge(2, 3);
        g.addEdge(3, 3);

        System.out.println("Following is Breadth First Search    " +
                "(starting from vertex 2)");

        g.BFS(2);
    }

    // if you want an analogy, take computer science explorers / software engineers

    // dfs:: some computer science explorers in life want to be experts in a particular branch, for example: networking
    // they go deeper into networking.. find its related concepts.. go deeper, till reaching a dead-end of researched areas.
    // after sometime, they might come back to visit other computer science topics.. this can be thought of depth first search

    // bfs:: for software engineers, (not all), we start with computer science..
    // visit its topics one at a time, during course of our career.
    // we dont go deep enough into one area. we touch multiple topics, in order to 'acquire' skills..
    // like: for eg: software engineering has: backend, front end, database, machine learning..
    // we might visit each of them , in order to 'acquire' these skills..
    // then later visit their related concepts.. ( this is a debatable analogy )

    // of course there are people who have T level ( breadth wise into multiple topics & depth into few topics )
}

    package main.hackerrank;
import java.io.*;
import java.math.*;
import java.security.*;
import java.text.*;
import java.util.*;
import java.util.concurrent.*;
import java.util.regex.*;

/**
 * https://www.hackerrank.com/challenges/counting-valleys/problem
 *
 * It must return an integer that denotes the number of valleys Gary traversed.
 */
public class CountingValleys {

    // Complete the countingValleys function below.

    /**
     * INPUT
     * 8
     * UDDDUDUU
     *
     * OUTPUT
     * 1
     *
     * INPUT
     *
     * 12
     * DDUUDDUDUUUD
     *
     * OUTPUT
     *
     * 2
     * @param n
     * @param s
     * @return
     */
    static int countingValleys(int n, String s) {

        // we can initialize with sea level , with value = 0;
        // when we go below 0, we have reached a valley.
        // once we cross negative, go to positive and then negative, we have reached another valley

        int level = 0;
        int valleyCount = 0;
        boolean prevValueNegative = false;
        char first = s.charAt(0);
        if(first == 'U'){
            level++;
        }
        else{
            level--;
        }

        if(level < 0){
            prevValueNegative = true;
        }

        


        for(int i=1; i< n; i++){

            if(level < 0){
                prevValueNegative = true;
            }
            else{
                prevValueNegative = false;
            }
            // if char is U, means, it is up
            // if char is D, means, it is down
            char inp = s.charAt(i);
            if(inp == 'U'){
                level++;
            }
            else
            {
                level--;
            }



            if(level == 0 && prevValueNegative){
                valleyCount++;
            }

            

        }

        return valleyCount;

    }

    private static final Scanner scanner = new Scanner(System.in);

    public static void main(String[] args) throws IOException {

        //BufferedWriter bufferedWriter = new BufferedWriter(new FileWriter(System.getenv("OUTPUT_PATH")));

        int n = scanner.nextInt();
        scanner.skip("(\r\n|[\n\r\u2028\u2029\u0085])?");

        String s = scanner.nextLine();

        int result = countingValleys(n, s);

        System.out.println(result);

        /*bufferedWriter.write(String.valueOf(result));
        bufferedWriter.newLine();
        bufferedWriter.close();
*/
        scanner.close();
    }
}

    Print characters and their frequencies in order of occurrence using a LinkedHashMap (java)

    1. Problem: Print characters and their frequencies in order of occurrence using a LinkedHashMap

     Input: str = “geeksforgeeks”
    Output: g2 e4 k2 s2 f1 o1 r1

     Here, we can see that, the output retains the order in which the characters appeared, so we can employ LinkedHashMap here.

     The logic can be seen as follows:
    - we can initialize a LinkedHashMap. The key will be Character, the value in the map will be Integer representing counter of how many times the character appeared in the input.
    - we will iterate over the input string and keep updating/ incrementing the count of characters depending on its occurrence.

    package main.geeksforgeeks.string;

import java.util.LinkedHashMap;

/**
 * https://www.geeksforgeeks.org/print-characters-and-their-frequencies-in-order-of-occurrence-using-a-linkedhashmap-in-java/
 *
 * Print characters and their frequencies in order of occurrence using a LinkedHashMap in Java
 *
 * Complexity: easy
 *
 * Logic is to use a linkedHashMap so that order of input characters are preserved.
 * And count of characters need to be incremented, whenever it appears again.
 */
public class CharFrequencyLHMap {

    public static void main(String[] args) {
        String str = "Integration Platform as a service";
        int n = str.length();
        printCharFrequency(str,n);
    }

    private static void printCharFrequency(String str, int n) {

        LinkedHashMap<Character, Integer> lhm = new LinkedHashMap<>();

        for(int i =0; i< n; i++){

            Character c= str.charAt(i);
            if(lhm.get(c) == null){
                lhm.put(c,1);
            }
            else{
                Integer currentCount = lhm.get(c);
                currentCount++;
                lhm.put(c,currentCount);
            }
        }

        for(Character c: lhm.keySet()){
            System.out.print(" " +c + lhm.get(c) + " ");
        }

    }
}

    How does CRED provide value ?



    Recently one of my friends referred me to this app. He said you can pay your credit card bills via this app and you can get rewarded for it.

    I was happy to experiment and he too was definitely happy as he was getting referral points.

    But doing that, I didn't imagine, might open doors to different perspectives of startups/ businesses.
    ( It gave me some insights about redemption model, listing fees, 'data localization' requirement of RBI etc )

     ( Data localization means that end to end transaction details need to be stored in India. Notification from RBI related to data localization. )

    3rd party businesses get listed on CRED platform. CRED might charge these businesses for listing. And whenever CRED user redeems a reward, CRED might make a margin for that transaction, as CRED is origin for that transaction.

     How does CRED provide value?

    Definitely on one hand, CRED is pulling credit worthy users. On the other hand, it's onboarding multiple businesses , even some of them could launch themselves via CRED.

     CRED is also looking at trust-building use-cases , for example, given that users are part of CRED.club or the 'credit-worthy' community, it would be easier for house-owner to trust new tenants, so that membership within CRED.club creates an implicit understanding that the tenant will pay rent regularly. The house-owner might also make decision to get rid of advance amount, since the tenant has proved to be trustworthy.

     As far as the CRED ecosystem is concerned, there are multiple players as shown in below picture:
    1. the users paying credit card bills via CRED app
    2.  merchants willing to provide offers to CRED users
    3. credit bureau providing details to CRED to access credit score of users
    4. banks / finance institutions to whom the bill payment is done


    Technically, looking at CRED app, there are multiple flows: namely: onboarding, transaction, redemption.

    For onboarding, acceptance of new members is done by accessing credit score.


    As part of transaction flow, users get 'coins' which could be redeemed.



    As part of redemption flow, users claim rewards from businesses.

    We can also think, CRED is a community based on trust and it could also be a distribution channel, as users inflow increases. For example, it can become a lending market-place and banks might reach CRED for their user-base.


    I am not completely aware of the business model of CRED, but curious though.

    System Design Architecture for Hotel Reservation System







    Photo by Francesca Saraco on Unsplash

     Let us consider the problem of developing an cloud-based application for hotel booking or reservations.

     What we mean is that:
    • users should be able to access our hotel booking application, check availability of rooms, make reservations, cancel reservations etc.
    • the application should be hosted over cloud, to reap benefits of cloud computing: namely: availability, scalability

     The goal of this article is to finally develop an architecture for this problem, considering cloud-based concepts.

     (
    We shall go with basic first principles perspective. We will deconstruct our problem into different parts ( indirectly ) to reach our goal
    )

     The fundamental things which we are aware :
    • Front-end
    • Back-end
    • Persistence

    This is a classic case of user-facing application.
    It presents a case for building up UI & corresponding backend.
    ( not all software systems require user interface )

     One thing to be clear is that, whenever we develop an application, we need to decide front-end platforms and back-end platforms for our application.

     Front-end:

     We can choose whether we are going to support different user-facing modes of the application.
    • Web application
      • Whether it makes sense for developing web-app. Nowadays, most of the end-user facing applications..
    • Mobile application
      • Android
      • iOS
    • Messaging interface
      • Facebook Messenger
    Back-end:

    While choosing the back-end part of the application, we can choose whether our services are going to be written in

    • Java 
    • Python 
    • Node.js
    • .. host of other languages...

    Coming back to our central focus: Architecture.

    courtesy: Peter Cripps

    As far as architectural thinking is concerned, there are three parts to it: What, Why & How.

    • What ?
      • What are the requirements of the system? 
      • What are the functional requirements? 
      • What are the nonfunctional requirements ?
      • What are the constraints for the system?
    • Why ?
      • Why did we make a decision to choose how a component is designed in a way?
      • Did we consider options to solve the requirements? 
      • Why did we choose one option over another? 
      • What are the pros and cons of each option?
        • For example, why should we use CDN approach similar to Netflix, to solve our problem?
    • How ?
      • What is the design for our system?
      • What is the implementation for our system?

    Let us come back to our system - hotel booking / reservation.

    What?
    Let us try to list the requirements for our system.

    Functional Requirements of system:

    1. Users should be able to login 
      1. This requires authentication
    2. Users should be able to view rooms availability
      1. For this, should we have a precondition that user should be logged in?
    3. Users should be able to make bookings
      1. For this, should we assume that, user should be logged in or in guest mode?
    4. Users should be able to cancel bookings
      1. What should be the validation mechanisms for cancellation?
    5. Users should be able to add this booking to their calendar - for example - integration with Google calendar / other calendar apps
      1. Should we support this feature?
    6. Users should be able to make payments seamlessly - this requires payment integrations
      1. Should we solve the problem of payment gateway or integrate with existing payment gateway?
    There are other miscellaneous functional requirements/ nice-to-have , which given the time, we can list endless. ( Feel free to add in comments with requirements related to our application )


    Having covered the What? , in the next sequence of articles, we shall focus on Why?, How? and finally arrive at our goal of defining our cloud-based architecture for hotel booking application.
